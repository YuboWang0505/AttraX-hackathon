# 文字 → 硬件控制规则

> 产品规则文档 · 描述 S/M 的聊天消息如何映射到硬件震动档位
> 实现代码：`backend/src/intent/pipeline.ts` + `backend/src/rooms.ts` + `frontend/src/pages/Chat.tsx`

## 一句话

**S 说话 → 档位跟着变；M 说话 → 档位不变；任一方说安全词 → 档位归 0 且会话结束。**

---

## 1. S 的文字 → 硬件

| S 说的内容 | 硬件响应 | 例句 |
|---|---|---|
| 安全词精确匹配 | **立即 0 档 + 会话终止** | "安全词"、"安全词！" |
| 前戏安抚 / 温柔话 / 闲聊 / 边缘控制否定句 | **1 档**（轻） | "抱抱"、"慢慢来"、"不准丢,忍住"、"今天天气不错" |
| 命令 / 挑逗 / 警告 / 轻度惩戒 | **2 档**（中） | "跪好"、"想要就求我"、"皮痒了是吧"、"罚你五下" |
| 高潮许可/强制 / 严厉惩罚 / 高度奖励 | **3 档**（强） | "丢吧"、"给我丢"、"狠狠惩罚你"、"表现真棒,全给你" |
| LLM 判断不明确 | **1 档**（默认兜底） | 随机闲聊、单字"嗯"、表情符 |
| LLM 网络/API 挂 | **1 档**（fallback） | 保证硬件不会因 AI 挂了就无响应 |

**核心规则：**

1. S 的每条消息**可以**改变档位（升、降、相同都行）
2. 档位变化是**瞬时**的 —— S 说完话 → 约 1 秒后 M 硬件响应
3. LLM **凭语义判断**，不是关键词匹配 —— "我现在给你倒杯水" 不会触发 3 档
4. S **不能直接让硬件归零**（不能说"停"就停）—— 归零只有安全词这一条路

---

## 2. M 的文字 → 硬件

| M 说的内容 | 硬件响应 |
|---|---|
| 安全词精确匹配 | **立即 0 档 + 会话终止** |
| 其它任何内容 | **档位保持不变**（沿用 S 上次定的值） |

**核心规则：**

1. M 说 "停"、"别动"、"轻点"、"求求你" —— **硬件全部不响应**
2. M 想让硬件停下来**只有一条路：说安全词**
3. 这是**故意的产品设计** —— SM 情境下的"求饶话术"和"真实停止请求"必须明确区分，安全词就是那条不可伪装的红线

---

## 3. 档位维持规则（时间维度）

```
假设会话开始档位 = 0

[t=0]   S:"乖,放松"         → LLM 判 1 档 → 档位 1，硬件轻震
[t=2]   M:"好舒服"          → 档位不变 = 1
[t=5]   M:"主人"            → 档位不变 = 1
[t=8]   S:"给我跪好"        → LLM 判 2 档 → 档位 2，硬件切到中震
[t=12]  S:"别动"            → LLM 判 2 档（命令语气）→ 档位仍 2
[t=15]  S:"给我丢"          → LLM 判 3 档 → 档位 3，硬件切到高震
[t=18]  M:"不行了"          → 档位不变 = 3
[t=20]  M:"安全词"          → 立即 0 档 + 会话终止
```

**关键观察：**

- **档位不会自动回落**。S 说 3 档之后不再发消息，档位一直停在 3 档
- 降档要 S 主动说一句软话（如 "乖" → 1 档）
- **没有"暂停但不终止"路径**。想停只有安全词 / peer_left / 退出按钮

---

## 4. 硬规则（不经 LLM）

不管 LLM 怎么判，以下路径**绝对优先**：

1. **安全词匹配** → 直接 0 档 + 终止会话
2. **WS 断开超过 30 秒** → 对方触发 `peer_left`，档位归 0 + 终止
3. **M 点"退出"按钮** → 本地 `writeIntensity(0)` + GATT 断开 + 页面 reload
4. **会话空闲 30 分钟** → 服务端广播 `peer_left`，档位归 0 + 终止

---

## 5. 档位到硬件的对照

| 档位 | 视觉（IntensityViz） | BT 写入字节 | ESP32 含义 |
|---|---|---|---|
| 0 | 静止 | `'3'` | 停 |
| 1 | 2 秒呼吸波纹 | `'2'` | 低 |
| 2 | 1 秒脉冲 | `'1'` | 中 |
| 3 | 0.3 秒高频脉冲 | `'0'` | 高 |

ESP32 协议是倒映射（`'0'=高` / `'3'=停`），由硬件固件定义，web 侧 `frontend/src/lib/bluetooth.ts:INTENSITY_TO_CMD` 做转换。

## 6. S 与 M 的能力对比

| 能力 | S | M |
|---|---|---|
| 调高档位 | ✅ 说对应话 | ❌ |
| 调低档位（非 0） | ✅ 说软话 | ❌ |
| 归 0 且继续会话 | ❌ | ❌ |
| 归 0 并终止会话 | ✅ 说安全词 | ✅ 说安全词 |
| 看到当前档位 | ✅ 视觉 | ✅ 视觉 + 身体 |
| 触发硬件 BT 配对 | ❌（S 侧跳过） | ✅（BtGate 页面） |
| 向硬件写档位 | ❌ | ✅ |

**隐含设计**：没有"归零但不终止"的路径。让硬件停下来，**必然伴随会话结束**。SM 场景里"不舒服到要停"不应该是轻量级操作，应该是明确的结束信号。若需要"暂停但不终止"，得另做 UI（比如 M 端"暂停"按钮 + 新消息类型广播）。

---

## 7. 会话结束时的硬件行为

不同结束路径对硬件的保证强度不同：

### 7.1 强保证路径（代码显式归零）

以下路径先执行 `bt.writeIntensity(0)` 再断 GATT 再 reload，**硬件必停**：

- 安全词（任一方触发）
- 对方 `peer_left`（断开超过 30s）
- 空闲超时 30 分钟
- M 点"退出"按钮
- BT 中断弹窗的"退出会话"按钮

### 7.2 弱保证路径（依赖 ESP32 固件）

以下路径浏览器**没有机会执行 JS 清理**，硬件是否停依赖 ESP32 固件的 BLE `onDisconnect` 回调是否把电机归零：

- 用户直接关标签页 / 关浏览器
- 浏览器崩溃 / 系统崩溃
- 手机锁屏超时后 BT 断连
- 走出有效距离 / WiFi 与 BT 冲突导致断连
- 电池耗尽

**参见 `HARDWARE-todo.md` 的要求项。** 硬件侧 `onDisconnect` 必须归零电机，否则"用户关机但硬件还在震"会成为真实事故。
